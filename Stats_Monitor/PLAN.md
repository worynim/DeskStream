# 📋 Stats_Monitor Project Plan

이 보고서는 PC/Mac의 시스템 정보를 실시간으로 수집하여 DeskStream 하드웨어로 전송하고 표시하는 프로젝트의 통합 계획을 담고 있습니다.

## 1. 프로젝트 개요
- **목표**: 상시 켜져 있는 외부 디스플레이를 통해 PC의 자원 사용량(CPU, GPU, RAM, SSD, 온도, 전력)을 모니터링함.
- **핵심 철학**: **"Dumb Display, Smart Client"**. 하드웨어는 들어오는 데이터를 렌더링만 하고, 모든 데이터 가공 및 로직은 파이썬 프로그램에서 처리하여 확장성을 극대화함.

## 2. 시스템 아키텍처

### 💻 PC/Mac 클라이언트 (Python)
- **운영체제**: Windows 10/11, macOS (Intel/Apple Silicon) 지원.
- **실행 방식**: 시스템 트레이(System Tray)에 상주하며 백그라운드에서 동작.
- **주요 라이브러리**:
    - `psutil`: CPU, RAM, SSD 정보 수집.
    - `bleak`: OS 통합 BLE(Bluetooth Low Energy) 통신.
    - `pystray`: 시스템 트레이 아이콘 및 메뉴.
    - `pyobjc` (macOS): Apple Silicon 온도 및 GPU 정보 추출.
    - `GPUtil` (Windows): NVIDIA GPU 정보 추출.

### 📡 통신 프로토콜 (Bluetooth BLE)
- **기술 선택**: 전력 효율과 최신 OS 호환성을 위해 **BLE (Bluetooth Low Energy)** 사용.
- **데이터 포맷**: **JSON (Minimized)**.
    - **장점**: 이기종 간 데이터 호환성 완벽, 필드 추가 및 변경이 매우 유연함.
    - **추천**: 필드명을 짧게 유지(e.g., `cpu_usage` -> `cu`)하여 통신 대역폭 최적화.

### 📟 하드웨어 (ESP32-C3)
- **역할**: BLE 서버로 대기하며, 수신된 JSON을 파싱하여 4개의 OLED에 데이터 매핑.
- **가변 레이아웃**: 특정 JSON 키가 들어오면 해당 화면의 내용을 동적으로 변경하는 구조.

## 3. 데이터 포맷 정의 (JSON Schema)
```json
{
  "c_u": 15,    // CPU Usage (%)
  "c_t": 48,    // CPU Temp (℃)
  "g_u": 20,    // GPU Usage (%)
  "g_t": 55,    // GPU Temp (℃)
  "r_u": 62,    // RAM Usage (%)
  "d_u": 40,    // Disk Usage (%)
  "p_w": 35.5   // Power (W)
}
```

## 4. 구현 단계별 계획

### 1단계: 파이썬 데이터 수집 모듈 개발
- OS별 하드웨어 센서 데이터 접근 권한 및 라이브러리 검증.
- 백그라운드 루프 및 트레이 아이콘 구현.

### 2단계: BLE 통신 구현
- ESP32-C3의 BLE Service/Characteristic 정의.
- 파이썬 클라이언트의 자동 재접속(Auto Reconnect) 로직 구현.

### 3단계: 하드웨어 다이나믹 렌더링 개발
- JSON 키-값 쌍을 화면 레이아웃에 매핑하는 매니저 클래스 작성.
- 4개 OLED의 갱신 속도 최적화.

### 4단계: 통합 테스트 및 품질 최적화
- 장시간 실행 시 메모리 누수 확인.
- 블루투스 연결 안정성 및 지연 시간(Latency) 최소화.

## 5. 기대 효과 및 확장성
- 하드웨어 펌웨어를 매번 수정하지 않고도 파이썬 코드만 수정하여 주식 정보, 날씨, 뉴스 헤드라인 등으로 화면 구성을 자유롭게 변경 가능.

## 6. 🚀 M2/M3 Mac 환경에서의 현재 상황 및 다음 단계 제안 (최근 진행 요약)

### 📝 지금까지 완료된 작업
1. **정밀 지표 매칭 완료**: `psutil`과 시스템 파티션을 분석하여 `mac stats` 앱과 동일한 RAM(83%) 및 Disk(60.2%) 수치를 산출하도록 튜닝을 성공적으로 마쳤습니다.
2. **GPU 실시간 수집**: `ioreg`를 통해 Apple Silicon GPU(AGXAccelerator)의 가동률 데이터를 추출하여 항상 0%로 나오던 문제를 해결했습니다.
3. **BLE 통신 및 백그라운드 안정화**: `send_data` 주기를 1초로 고정하고, `powermetrics` 등의 터미널 도구를 스레드로 분리하여 블로킹 없는 비동기 전송망을 완성했습니다. 아두이노 측 디버그 로직도 마련했습니다.

### ⚠️ Mac 하드웨어(SMC) 한계 봉착
`mac stats` 앱처럼 '시스템 전체 전력(W)'과 '온도'를 **1초 단위로 부드럽게** 가져오는 것은 Python 백그라운드 스크립트(`powermetrics` / `ioreg` 명령어)만으로는 한계가 있음이 확인되었습니다.
* **전력(W)**: `powermetrics`는 1초마다 갱신하지만 "CPU/GPU 칩 자체 전력(약 6W)"만 리포트하고, `ioreg`는 "시스템 전체 전력(약 20W)"을 보여주지만 애플 정책상 갱신 주기가 10~30초로 매우 깁니다. 결과적으로 **(1초 반응형 6W)** vs **(느린 20W)**의 딜레마가 발생했습니다.
* **온도(℃)**: Apple Silicon 기반 데스크탑(Mac mini/Studio 등)은 랩탑과 발열 관리 트리거가 달라, 터미널 명령어(`osx-cpu-temp`, `powermetrics`)에서 온도 출력을 컴파일 단에서 막아두었습니다 (항상 0.0℃ 반환). 

### 💡 다음 세션 진행을 위한 제안 (선택 필요)
다음 세션을 시작할 때, 하드웨어적 한계를 고려하여 파이썬 클라이언트와 OLED 화면을 어떤 방향으로 마무리할지 결정하고 진행합니다.

1. **옵션 A (가젯의 생동감 우선 - 가장 추천)**
   * 전력을 1초마다 빠르게 반응하는 **"순수 CPU/GPU 소비전력(~6W)"**으로 채택하여 화면이 멈춰있지 않게 합니다.
   * 작동하지 않는 온도는 화면에서 제거하고, 빈 자리에 **"실시간 네트워크 다운로드/업로드 속도(Mbps)"**를 추가해 OLED 모니터의 생동감을 높입니다.
2. **옵션 B (시스템 총 전력 중시)**
   * 전력을 **"시스템 전체 소비전력(~20W)"**으로 채택하되, 값이 10~20초에 한 번만 바뀐다는 한계를 그대로 수용합니다. 온도는 0으로 남겨둡니다.
3. **옵션 C (C++/Swift 네이티브 브릿지 개발 - 고난이도)**
   * 애플의 비공개 IOHID API를 직접 파싱하는 C 레벨의 프로그램을 새로 만들어서 파이썬과 연동합니다. (맥 보안 권한 문제 등 변수가 많아 작업 난이도가 급상승합니다.)
